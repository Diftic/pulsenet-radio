# Red-Team Security Review: PulseNet Player

- **Date:** 2026-04-30
- **Target:** `D:\PycharmProjects\pulsenet-player`, branch `master`, version `1.6.3` (working tree)
- **Scope:** Local code review with theoretical exploit chains. No active testing against GitHub, YouTube, or any external service.
- **Method:** Six parallel attack subagents (Opus), one per surface: web→native bridge, auto-update path, native interop / threading, settings & filesystem, installer & CI, process lifecycle & privacy.
- **Outcome:** **Two CRITICAL findings, one of which is currently exploitable end-to-end.** Several HIGH-severity issues clustering around the auto-update path. The hotkey hook, audio bridge memory safety, JSON deserializer, virtual host path-traversal defense, and the (absent) telemetry posture all came out clean.

---

## 0. Threat-model filter (open-source desktop app)

PulseNet is an open-source Windows desktop app. The realistic attacker model is:

- **In scope:** remote / supply-chain / network-surface attacks. Anything an external party can do without already having user-context code execution on the victim's machine. Maintainer account compromise, CI token leakage, upstream action / WiX compromise, MITM via a trusted-root cert, malicious XSS reaching the renderer.
- **Out of scope:** local same-user attacks. Once arbitrary user-mode code is running on the machine, every desktop app on the box is exploitable in roughly the same way (read browser cookies, swap binaries on disk, drop autorun entries, synthesize tray clicks, race files in `%TEMP%`). Hardening PulseNet against same-user malware is academic given the rest of the desktop's attack surface.

After applying that filter to the full report below:

**Still critical:**
- **CRITICAL-1** (unsigned auto-update, no client-side verification). Remote attack. Highest blast radius.

**Still high (all supply-chain):**
- **HIGH-3** (floating GitHub Action major-tag pins).
- **HIGH-4** (floating `wix --version "5.*"`).
- **HIGH-5** (`<Files Include="stage\**" />` glob).

**Still worth fixing:**
- **MEDIUM-1** (no `e.Source` validation in `WebMessageReceived`). One-`if`-statement defense-in-depth against future renderer compromise.
- **MEDIUM-3** (no host allow-list in `OpenInDefaultBrowser`). Phishing handoff if the renderer is ever XSS'd.
- **MEDIUM-10** (no `dependabot.yml` / `NuGet.config` source pinning). Supply-chain hygiene.
- **L-7** (version-spoof rollback) and **L-8** (no HTTPS cert pinning). Both closed automatically by the CRITICAL-1 fix; signature-on-artifact is stronger than channel pinning.
- **L-13** (`<Deterministic>true</Deterministic>` + SLSA provenance). Lets users independently verify a release came from the public source.

**Drops out (accepted as OSS-desktop reality):**
- CRITICAL-2 (MsiPath retry), HIGH-1 (TOCTOU on update temp dir), HIGH-2 (PowerShell trampoline race), HIGH-6 (mutex squat), MEDIUM-2 (StealForeground focus theft) all require same-user code execution.
- MEDIUM-4 through MEDIUM-9 are reliability / defense-in-depth, not exploits.
- Most of the L-list (debug logs, banner offscreen clamp, app.manifest typo, etc.) is quality-of-life.

These findings remain documented in the sections below for completeness, but they are not gating items for public distribution under this threat model.

## Executive summary

The single most important finding is that **the auto-updater downloads an unsigned MSI from GitHub Releases and runs it via `msiexec` with no signature, hash, publisher, or Authenticode verification of any kind**. A compromised maintainer GitHub account, a leaked CI token, a corporate-MITM proxy with a trusted-root cert, or a compromise of the floating `softprops/action-gh-release@v2` action all collapse into the same outcome: **arbitrary code execution as the user, on every PulseNet install, on next launch**. The source code at `src/Services/SelfUpdateService.cs:107-110` openly acknowledges the MSI is unsigned (the team's response to AV detection was to switch from `/passive` to `/qb` to look more user-initiated). This is the largest blast-radius issue in the codebase and, combined with the supply-chain hygiene gaps in CI (HIGH-3, HIGH-4), should be the priority-1 fix before further public distribution.

The second critical issue is local in scope but immediately exploitable: the tray-icon "Retry failed update" flow trusts a path stored in a user-writable JSON file at `%LocalAppData%\pulsenet-radio\update.status.json` and feeds it directly to `Process.Start(..., UseShellExecute=true)`. Any same-user process can drop a status file pointing at any executable on disk; PulseNet itself surfaces a tray balloon urging the user to click Retry. This is a clean social-engineering primitive for malware that already has user-context execution to gain a more "trusted" path.

Several MEDIUM findings document defensive gaps that aren't immediately exploitable but reduce defense-in-depth: the WebView2 message bridge does not validate `e.Source`, the foreground-stealing `StealForeground` + `AllowSetForegroundWindow(ASFW_ANY)` pair can be triggered during in-session security prompts, the AudioBridge has a few COM-handle leaks across `RunOnce` iterations, and the single-instance mutex is squattable.

The cleared-list (Section 5) is meaningful: the keyboard hook does not log or persist keystrokes, no telemetry exists, the JSON deserializer is configured safely, the `YoutubeChannelId` setting is URL-encoded, and the virtual host path-traversal defense at `OverlayWindow.xaml.cs:1023-1025` is correctly placed.

## 1. Critical findings

### CRITICAL-1 — Unsigned auto-update MSI with no client-side verification

**Severity:** Critical &nbsp;|&nbsp; **Surface:** Update path + Installer/CI &nbsp;|&nbsp; **CWE:** 494, 345, 347

**Where:**
- `src/Services/SelfUpdateService.cs:79-198` (`ApplyMsiAsync`) and `:203-248` (`ApplyPortableAsync`) - download, no verify, then `msiexec /i`.
- `src/Services/SelfUpdateService.cs:107-110` - source comment confirms the MSI is unsigned: `"/passive was flagged heuristically by Bitdefender / Malwarebytes on our unsigned MSI; /qb shows a visible progress window".`
- `.github/workflows/build.yml:1-71` - no signtool / Authenticode / SBOM step. Builds and uploads as-is.
- `src/Services/UpdateChecker.cs:14` - update URL is the public GitHub Releases API for `Diftic/PulseNet-Player/releases/latest`.

**What:**
The download in `SelfUpdateService.cs:91-97` is `await _http.GetAsync(msiUrl) → File.Create(tempMsi) → CopyTo`, then a generated PowerShell script at `:151` runs `Start-Process msiexec.exe -ArgumentList '/i "<tempMsi>" /qb /norestart ...' -Wait`. Repo-wide grep for `WinVerifyTrust|Authenticode|X509|sha256|signature` in update code returns **zero hits**. There is also no `SocketsHttpHandler.SslOptions.RemoteCertificateValidationCallback` set, so any cert chaining to a Windows-trusted root is accepted (corporate MITM proxies, malicious enterprise CAs).

**Exploit paths:**
1. **Maintainer account / PAT compromise** - one push of a `v*` tag to `Diftic/PulseNet-Player` produces a `draft: false` release that every running client polls and consumes within minutes (`UpdateChecker.cs` runs on startup and via the manual button).
2. **Compromise of `softprops/action-gh-release@v2`** (floating major-tag pin in `build.yml:63`) - attacker code runs inside the build job with `contents: write` and substitutes the uploaded MSI.
3. **MITM with a trusted-root cert** - any party with a cert in a victim's Windows root store substitutes both the JSON metadata and the asset bytes during `_http.GetAsync`.
4. **Floating `wix --version "5.*"`** at `build.yml:45` - any future malicious 5.x release of WiX runs at MSI generation time, on the runner, with the `GITHUB_TOKEN`.

**Impact:** RCE as the user, on every install. Per-user MSI scope (`installer/installer.wxs:11`) means user context, not SYSTEM, but the malicious MSI controls all install actions: drops binaries, registers Run-keys / shortcuts, runs custom actions, then `Start-Process $exe` (`SelfUpdateService.cs:166`) auto-launches the new exe. Persistence on every install of the app, in one push.

**Fix (layered, all required):**
1. **Sign the MSI in CI** - Authenticode via Trusted Signing or a real EV cert. Gate behind a GitHub `environment:` with required reviewers so a leaked workflow token cannot sign on its own.
2. **Verify on the client before `msiexec`** - call `WinVerifyTrust` (`WINTRUST_ACTION_GENERIC_VERIFY_V2`) and pin the certificate subject or thumbprint:
   ```csharp
   if (!AuthenticodeVerifier.VerifyEmbeddedSignature(tempMsi))
       throw new SecurityException("MSI signature invalid");
   if (!AuthenticodeVerifier.SubjectMatches(tempMsi, "CN=PulseNet Player, ..."))
       throw new SecurityException("Unexpected publisher");
   ```
3. **Pin the digest** - publish a SHA-256 in the release notes and verify in `CheckAsync`. Defense-in-depth alongside (2).
4. **Pin every GitHub Action by full commit SHA** (`actions/checkout@<sha>`, `actions/setup-dotnet@<sha>`, `softprops/action-gh-release@<sha>`).
5. **Pin `wix` exactly:** `dotnet tool install --global wix --version 5.0.2`.
6. **Replace `<Files Include="stage\**" />`** at `installer/installer.wxs:42` with an explicit per-file `<File>` list, and add a CI step that fails the build if `installer\stage\` contains anything not on an allowlist.
7. **Set `AllowAutoRedirect=false`** on the update `HttpClient` and follow redirects manually with host validation.
8. **Drop the staged PowerShell trampoline** at `SelfUpdateService.cs:121-186` - inline the wait-for-pid + msiexec in a small detached helper exe, signed and shipped with the app, instead of writing a `.ps1` to a predictable temp path.

### CRITICAL-2 — Tray "Retry failed update" runs `Process.Start(UseShellExecute=true)` on a path read from a user-writable JSON file

**Severity:** Critical (locally exploitable today) &nbsp;|&nbsp; **Surface:** Settings + Process &nbsp;|&nbsp; **CWE:** 73, 426, 494

**Where:**
- `src/Services/SelfUpdateService.cs:33-45` - `ReadLastResult()` deserializes `%LocalAppData%\pulsenet-radio\update.status.json` with no validation of the `MsiPath` field.
- `src/App.xaml.cs:76-85` - reads it on startup and unconditionally calls `_trayIcon.SetPendingUpdateRetry(lastUpdate.MsiPath)` whenever `success == false`.
- `src/UI/TrayIcon.cs:50-57, 59-75` - `OnRetryUpdate` runs `Process.Start(new ProcessStartInfo(_pendingMsiPath) { UseShellExecute = true })` after a `File.Exists` check and nothing else.

**Exploit:**
Any process running as the same user drops:
```json
{ "timestamp":"2026-01-01T00:00:00Z","success":false,"exitCode":1603,
  "msiPath":"C:\\Users\\victim\\AppData\\Roaming\\evil\\payload.exe" }
```
On next launch:
1. `App.xaml.cs:76` parses the file - `MsiPath` is now attacker-controlled.
2. `App.xaml.cs:79` arms the tray menu's Retry item.
3. `App.xaml.cs:80-83` shows a balloon: *"Update didn't install - Right-click the tray icon → Retry failed update."* The application itself surfaces the social-engineering trigger.
4. User right-clicks tray and selects Retry.
5. `TrayIcon.cs:69` shell-executes the attacker-controlled path.

`UseShellExecute=true` will dispatch `.exe`, `.bat`, `.cmd`, `.lnk`, `.cpl`, `.scr`, etc., not just MSIs. There is no extension check, no path-prefix check, no signature check.

Note: `App.xaml.cs:84` calls `ClearLastResult()` on disk after surfacing the retry, but `_pendingMsiPath` is preserved in memory for the rest of the session.

**Fix (apply all):**
1. Validate the path before arming retry:
   ```csharp
   var expectedRoot = Path.Combine(Path.GetTempPath(), "pulsenet_update");
   var fullPath = Path.GetFullPath(lastUpdate.MsiPath);
   if (!fullPath.StartsWith(expectedRoot, StringComparison.OrdinalIgnoreCase) ||
       !fullPath.EndsWith(".msi", StringComparison.OrdinalIgnoreCase))
       return;
   ```
2. Or better, **don't persist the path at all**. Persist only the failure reason; on retry, re-run `UpdateChecker.CheckAsync` and download fresh. The path is deterministic from `Path.Combine(Path.GetTempPath(), "pulsenet_update", "pulsenet_update.msi")` anyway.
3. Invoke `msiexec.exe /i "<path>" /qb /norestart` explicitly (`UseShellExecute=false`), not `ShellExecute` on the path, to remove the arbitrary-extension footgun.
4. Verify the cached MSI's signature (CRITICAL-1's fix) before launching.
5. Clear `_pendingMsiPath` when `ClearLastResult()` runs.

## 2. High findings

### HIGH-1 — Predictable temp path: TOCTOU between MSI download and `msiexec /i`

**Where:** `src/Services/SelfUpdateService.cs:87-89, 96, 151` (`%TEMP%\pulsenet_update\pulsenet_update.msi`)

**What:** The temp directory and MSI filename are static constants. Same-user malware can race-replace `pulsenet_update.msi` between download completion and `msiexec` invocation (the wait-for-PID-exit loop at `SelfUpdateService.cs:139-142` extends the window up to ~30 s). Hard-link / junction tricks are also possible if the directory or file are pre-created with permissive ACLs.

**Fix:** Use `Path.GetRandomFileName()` for the MSI; create the temp dir under `%LocalAppData%\pulsenet-radio\update-<guid>\` with restrictive ACL; verify signature **inside** the launcher script immediately before `Start-Process msiexec`, not in the C# code that hands the path off.

### HIGH-2 — PowerShell trampoline script written to predictable path

**Where:** `src/Services/SelfUpdateService.cs:185-186, 189` (`%TEMP%\pulsenet_update\update.ps1` then `powershell.exe -ExecutionPolicy Bypass -File "<scriptPath>"`)

**What:** Same-user attacker who races between `WriteAllTextAsync` and `Process.Start` swaps the script. PulseNet runs it with `-ExecutionPolicy Bypass`. The `EscapeSingleQuoted` function at `:201` is correct for its job, but the *delivery channel* (predictably-named script file written to `%TEMP%`) is the actual weakness.

**Fix:** Pass the script via `-EncodedCommand <base64>` instead of `-File` so there is no file an attacker can swap. Or drop PowerShell entirely.

### HIGH-3 — GitHub Actions pinned to floating major tags

**Where:** `.github/workflows/build.yml:15, 23, 63` (`actions/checkout@v4`, `actions/setup-dotnet@v4`, `softprops/action-gh-release@v2`)

**What:** Floating major-tag refs resolve to whatever commit the maintainer of the action currently points at. The `softprops` action is third-party and historically the highest-risk class - a successful compromise yields code execution inside the build job with `contents: write` and the `GITHUB_TOKEN`.

**Fix:** Pin every action by full commit SHA. Add `.github/dependabot.yml` for `package-ecosystem: github-actions`.

### HIGH-4 — `wix --version "5.*"` floats both minor and patch

**Where:** `.github/workflows/build.yml:45`

**What:** Any future malicious 5.x WiX release executes during MSI generation on the runner.

**Fix:** Pin exactly: `dotnet tool install --global wix --version 5.0.2`.

### HIGH-5 — `<Files Include="stage\**" />` packages everything dropped into the stage dir

**Where:** `installer/installer.wxs:42`, `.github/workflows/build.yml:50-51`

**What:** Any file the publish step or a transitive build target drops into `installer\stage\` is silently shipped in the MSI. Combined with CRITICAL-1, the user has no way to see the difference. `Microsoft.Windows.CsWin32` declares `IncludeAssets=...; build; ...` which broadens the build trust boundary.

**Fix:** Replace the glob with an explicit `<File Source="stage\..." />` list or a checked manifest. Add a CI step that fails if `stage\` contains anything off-allowlist.

### HIGH-6 — Single-instance mutex is squattable, silently denying launches

**Where:** `src/Program.cs:19-24`, `src/Constants.cs:6` (`MutexId` = compile-time constant exposed in the public repo)

**What:** `new Mutex(true, "pulsenet-3C8F4A2D-...")` uses the local-session namespace by default. Any process running in the same session can pre-create the mutex with that exact name (read it from the public source), hold it forever, and PulseNet exits silently every launch with no balloon and no log (logger isn't constructed yet). Five-line "blocker" exe pinned to startup is a permanent kill.

The `if (!isNewInstance) return;` at `Program.cs:23` also swallows legitimate second-launch attempts with no surface message.

**Fix:**
- `Local\pulsenet-...` namespace prefix and a `MutexSecurity` ACL pinned to the current user SID.
- Implement the TODO at `Program.cs:22` to broadcast a registered window message to the existing instance so the user knows the second launch was redirected.
- On `AbandonedMutexException`, take ownership.

## 3. Medium findings

### MEDIUM-1 — `WebMessageReceived` does not validate `e.Source`

**Where:** `src/UI/OverlayWindow.xaml.cs:378-389`, `src/UI/MiniBannerWindow.xaml.cs:233-269`

**What:** Both handlers read the message body without checking the originating frame URL. WebView2's `args.Source` is available but unused. Today no XSS path into `pulsenet.local` is demonstrable - the renderer doesn't `innerHTML` any string from messages, settings, or URL params - so the gap is defensive. **But:** every privileged case (`checkForUpdates`, `hotkey`, `streamerMode`, `minimizeMode`, `lock`, etc.) becomes attacker-reachable the moment any future change introduces a same-origin script that an attacker can influence.

**Fix:**
```csharp
if (!Uri.TryCreate(e.Source, UriKind.Absolute, out var src) ||
    !string.Equals(src.Host, Constants.PlayerVirtualHost, StringComparison.OrdinalIgnoreCase) ||
    !string.Equals(src.Scheme, "https", StringComparison.OrdinalIgnoreCase))
{
    return;
}
```
Also tighten `IsAllowedOrigin` (`OverlayWindow.xaml.cs:1211`) to require `Scheme == "https"`, not just host match.

### MEDIUM-2 — `StealForeground` + `AllowSetForegroundWindow(ASFW_ANY)` is a focus-takeover primitive

**Where:** `src/UI/OverlayWindow.xaml.cs:303-323`, `src/Services/GlobalHotkeyListener.cs:159`

**What:** `StealForeground` calls `AttachThreadInput(fgThread, uiThread, true)` then `BringWindowToTop` then detaches. Paired with `AllowSetForegroundWindow(0xFFFFFFFF)` in the LL keyboard hook. Result: pressing F9 while a security-relevant in-session prompt is foreground (browser permission prompt, MOTW SmartScreen, password manager autofill prompt) yanks the overlay in front of it. UAC consent is on the secure desktop (immune); other in-session prompts are not.

**Fix:** Replace with `SetWindowPos(HWND_TOPMOST, SWP_NOACTIVATE)` plus `FlashWindowEx` if attention is needed. Drop the `ASFW_ANY` call.

### MEDIUM-3 — `openUrl` web-message is a renderer→shell handoff

**Where:** `src/UI/OverlayWindow.xaml.cs:471-473, 1201-1209`

**What:** Renderer can post `{"type":"openUrl","url":"<x>"}` and the host calls `Process.Start(uri, UseShellExecute=true)`. Scheme is allow-listed to `http(s)://` only (good - blocks `file://`, `javascript://`, `ms-settings:`). Host is not allow-listed, so any HTTP/S URL opens in the user's default browser - a clean phishing-handoff primitive if MEDIUM-1 is paired with any future XSS.

**Fix:** Allow-list known hosts (`youtube.com`, `discord.com`, `github.com/Diftic/PulseNet-Player`, etc.); reject everything else.

### MEDIUM-4 — AudioBridge COM / event-handle leaks across `RunOnce` iterations

**Where:** `src/Services/AudioBridge.cs:104-321` (the `finally` at `:308` doesn't release `IMMDeviceEnumerator`, `IMMDevice ppEndpoint`, `IActivateAudioInterfaceAsyncOperation op`, the `ManualResetEvent` inside the activation handler).

**What:** Each toggle of Streamer Mode causes a `RunOnce` invocation; references accumulate until GC. Not a security-critical vector, but combined with the WebView2-driven `streamerMode` web-message it becomes a slow-DoS surface (renderer toggles fast → handles leak → audio service eventually refuses new activations).

**Fix:** Hoist `op`, `renderDevice`, `enumerator` into the outer scope and release them in the `finally`. Dispose the activation handler's `ManualResetEvent`.

### MEDIUM-5 — AudioBridge does not honor cancellation token during 5 s activation wait

**Where:** `src/Services/AudioBridge.cs:359` (`handler.Done.WaitOne(5000)`)

**What:** App shutdown can hang up to 5 s if the audio activation is in flight. After `_pumpThread.Join(2000)` times out, the host disposes while the pump is still in unmanaged code. Possibility of finalizer-thread crash on already-disposed CTS or COM objects.

**Fix:** Register `token.Register(() => handler.Done.Set());` after creating the handler. Replace the bare `WaitForSingleObject(captureEvent, 200)` with `WaitHandle.WaitAny([captureEvent, token.WaitHandle], 200)`.

### MEDIUM-6 — Hook callbacks marshal onto UI thread without exception protection

**Where:** `src/UI/OverlayWindow.xaml.cs:730, 832`

**What:** `MouseHookProc` and `KeyboardHookProc` perform unmanaged-to-managed work (`Marshal.PtrToStructure`, `Dispatcher.InvokeAsync`, `WebView.CoreWebView2.ExecuteScriptAsync`) without a top-level `try/catch`. An exception escaping into Windows' kernel hook dispatcher is undefined behavior. Slow callbacks also risk being silently dropped by Windows' `LowLevelHooksTimeout` (default 300 ms), with no observability.

**Fix:** Wrap each hook proc in `try { ... } catch (Exception ex) { _logger.LogDebug(ex, "hook proc"); }` before `CallNextHookEx`. Move heavy work to a worker thread driven by a `Channel`/`BlockingCollection` filled from the hook.

### MEDIUM-7 — `App.OnExit` is `async void`; awaitable shutdown races with process termination

**Where:** `src/App.xaml.cs:145`

**What:** WPF doesn't await `OnExit`. The continuation after `await _host.StopAsync(...)` can run after the runtime begins teardown, potentially during AudioBridge cleanup, and before the host's logger pipeline is gone. Intermittent shutdown crashes possible; potential corrupted settings file mid-write.

**Fix:** Block synchronously: `_host?.StopAsync(TimeSpan.FromSeconds(5)).GetAwaiter().GetResult();` or move awaitable work to `OnSessionEnding` and keep `OnExit` synchronous.

### MEDIUM-8 — Banner offscreen-lockout via poisoned `BannerLeft`/`BannerTop`; tray menu doesn't expose banner reset

**Where:** `src/UI/MiniBannerWindow.xaml.cs:275-291`, `src/UI/TrayIcon.cs:36`

**What:** Saved coords are applied directly with no on-screen-rect intersection. `Reset Player Window` in the tray menu only resets the overlay; banner reset is reachable only inside the in-overlay Miniplayer Settings sub-panel.

**Fix:** In `AnchorToSavedOrDefault`, intersect with `SystemParameters.VirtualScreen` and fall back to default if zero overlap. Add a `Reset Banner Position` tray menu item.

### MEDIUM-9 — `BannerOpacity` and `BannerScalePct` not clamped on read

**Where:** `src/Models/PulsenetSettings.cs:21-22`, `src/UI/MiniBannerWindow.xaml.cs:44-46`

**What:** Settings file with extreme values (`BannerScalePct: 1000000`, negative numbers) crashes the constructor before WPF's dispatcher catches it. App fails to launch; tray icon never wires up; no recovery path.

**Fix:** Clamp in `SettingsManager.Load` after deserialization for `BannerScalePct`, `BannerOpacity`, `WebViewZoomPct`, and any future user-tunable numerical setting.

### MEDIUM-10 — No Dependabot, no `NuGet.config` source pinning, no `<PrivateAssets>` discipline

**Where:** Missing `.github/dependabot.yml`, missing `NuGet.config`, `src/pulsenet.csproj:26-34` (no `<PrivateAssets>` on shippable deps).

**What:** Latent dependency-confusion risk if a private feed is ever added. No automated alerts on supply-chain CVEs. Ship-only-runtime discipline missing.

**Fix:** Add `NuGet.config` with `packageSourceMapping` pinned to `nuget.org`. Add `dependabot.yml` for `nuget` and `github-actions`. Audit `<PackageReference>` to apply `<PrivateAssets>all</PrivateAssets>` everywhere except the runtime-shipped DLLs.

## 5. Cleared (verified safe)

These hypotheses were investigated and refuted by code review. Worth recording so they don't get re-investigated next pass:

- **No keystroke leak.** End-to-end trace: `WH_KEYBOARD_LL` callback → `_heldKeys` (HashSet of VK codes) → `HotkeyPressed` event → no file/network/serialization. Comment at `GlobalHotkeyListener.cs:113` explicitly says "Always pass to the next hook - never swallow keys." The overlay's `KeyboardHookProc` only forwards an action *name* (one of 7 hard-coded literals: `space`, `mute`, `left`, `right`, `up`, `down`), never the original character.
- **No telemetry / phone-home.** Exhaustive grep: only two `HttpClient` instances, both pointed at GitHub Releases. Renderer has no `fetch`, `XMLHttpRequest`, `sendBeacon`, `gtag`, `analytics`, `telemetry` calls. `User-Agent` is `PulseNet-Player/{Maj.Min.Build}` only - no `InstallationId`, no machine name, no Windows version.
- **JSON deserializer is safe.** `System.Text.Json` doesn't support `TypeNameHandling`-style polymorphism. No custom `JsonConverter`s touch `Type` / `Activator.CreateInstance`. `KeyboardShortcut.JsonConverter` reads a fixed `KeyboardKey[]` and constructs the type directly. Unknown enum values throw `JsonException` and fall back to defaults.
- **Virtual host path traversal defended.** `OnRendererResourceRequested` (`OverlayWindow.xaml.cs:1023-1025`) checks `filePath.StartsWith(rendererFolder, …)` after `Path.Combine`. Requests like `https://pulsenet.local/../../Windows/win.ini` are rejected.
- **`openUrl` scheme allow-list works.** Only `http(s)://` accepted, blocking `file:`, `javascript:`, `ms-settings:`, `cmd:`, etc.
- **Navigation origin allow-listed.** `OnNavigationStarting` cancels everything except `https://pulsenet.local/` in the main frame.
- **Iframe permissions are minimal.** `iframe.setAttribute('allow', 'autoplay; encrypted-media; picture-in-picture')`. No `clipboard-write`, `geolocation`, `camera/microphone`, `display-capture`.
- **YouTube iframe cannot reach `chrome.webview`.** Cross-origin frames don't get the host-object exposure by default. No `AdditionalAllowedFrameAncestors` widening, no `AreHostObjectsAllowedInFrame`. No `window.addEventListener('message', …)` proxy in `player.js`.
- **`Process.Start` only with renderer-controlled string is `openUrl`** (scheme-restricted) and `checkForUpdates` (which uses `info.MsiUrl` from the GitHub API JSON - not from the web message).
- **`AddScriptToExecuteOnDocumentCreatedAsync` is safe.** Single use - sets `window.__pulsenetVersion = <jsonSerializedAssemblyVersion>;`. Compile-time value, JSON-encoded.
- **`Buffer.MemoryCopy` bounds correct.** `(long)framesToWrite * frameSize` widens before multiplication; clamping at `AudioBridge.cs:269` prevents oversized writes; pointers null-checked.
- **STA/MTA discipline correct.** WASAPI types are agile (`Both`); AudioBridge never marshals into UI objects from the pump thread.
- **WiX has zero `<CustomAction>` elements.** No scripts, no exes, no registry writes with attacker-influenced data at install time.
- **No service installation, no scheduled task, no Run-key writes** in the legitimate MSI.
- **No DLL search-order hijack primitive.** All `[DllImport]` references name OS DLLs (`user32.dll`, `kernel32.dll`, `dwmapi.dll`, `mmdevapi.dll`, `avrt.dll`, `ole32.dll`). No name-only `LoadLibrary` of a non-system DLL.
- **Per-user MSI scope** - no SYSTEM-context LPE primitive from the install.
- **HTTPS enforced** (`UpdateChecker.cs:14` is a `const`, no `http://` fallback, not settings-derived).
- **Strict version comparison** prevents auto-downgrade.
- **No msiexec command-line injection** - args are static format string with two app-controlled paths.
- **`YoutubeChannelId` is URL-encoded** before being placed in the player URL (`OverlayWindow.xaml.cs:296`).
- **`Mutex` cross-user namespacing OK.** `%APPDATA%` is per-user; cross-user squat doesn't apply (within-session same-user squat does - HIGH-6).
- **WebView2 cache folders inherit per-user `%APPDATA%` ACL.** No code path widens permissions.

## 6. Attack chains

### Chain A - "One push, all users compromised" (CRITICAL)
1. Maintainer GitHub PAT phished, or `softprops/action-gh-release@v2` or `wix --version "5.*"` compromised upstream (HIGH-3, HIGH-4).
2. Malicious `v*` tag pushed → CI builds and uploads unsigned MSI.
3. Every running PulseNet client polls `releases/latest` on next startup or via the manual button.
4. Client downloads the MSI (CRITICAL-1: no signature, no hash, no publisher pin) and `msiexec`'s it.
5. RCE as the user, persistence via Run-keys, on every install.

### Chain B - "User-context malware launders trust through PulseNet" (CRITICAL)
1. Malware running as the user drops `%LocalAppData%\pulsenet-radio\update.status.json` with `MsiPath` pointing at its own EXE.
2. Drops the payload at the matching path.
3. User launches PulseNet; tray balloon: *"Update didn't install - Right-click → Retry failed update."*
4. User clicks Retry. (CRITICAL-2: `Process.Start(UseShellExecute=true)` - any registered file type.)
5. The malicious payload runs from a trusted-feeling UI affordance. Defeats user vigilance.

### Chain C - "Same-user race on update temp dir" (HIGH)
1. PulseNet downloads MSI to `%TEMP%\pulsenet_update\pulsenet_update.msi`.
2. Same-user attacker, watching the predictable path (HIGH-1), races between the C# `File.Create` and the wait-for-PID-then-msiexec window (~30 s).
3. Swaps the MSI bytes (or, simpler: swaps `update.ps1` per HIGH-2).
4. msiexec runs the attacker MSI.
5. Defeats *future* signature-checks placed in C# code; only signature-check inside the PowerShell trampoline survives.

### Chain D - "Mutex squat + status-JSON drop" (HIGH)
1. Malicious helper holds the well-known mutex name (HIGH-6).
2. PulseNet refuses to launch silently; user retries several times.
3. User opens Task Manager, sees nothing wrong, gives up.
4. Helper drops a malicious `update.status.json` (Chain B), releases the mutex.
5. User finally launches PulseNet, sees the "Retry failed update" prompt, clicks.

### Chain E - "WebView2 message bridge + future XSS"
1. Future renderer change introduces an XSS in `pulsenet.local` (e.g., a string interpolated into `innerHTML`).
2. (MEDIUM-1) `OnWebMessageReceived` doesn't validate `e.Source`, so attacker JS posts arbitrary messages.
3. Attacker silently changes `streamerMode`, rebinds the hotkey to a key the user never presses, calls `openUrl` for phishing.
4. Cannot directly trigger update-RCE because `info.MsiUrl` comes from the GitHub API, not from the renderer message - **chain currently breaks**.
5. If GitHub release also compromised (Chain A), the chains merge into universal RCE.

## 8. Prioritized remediation roadmap

Filtered for the OSS-desktop threat model from Section 0. Items not listed here are documented in Sections 1-4 but are not gating under this threat model.

**Block-public-release tier (supply chain + remote):**
1. **Sign the MSI in CI** (Authenticode via Trusted Signing or EV cert) and **verify the signature on the client** before `msiexec`. Closes CRITICAL-1, L-7, L-8.
2. **Pin every GitHub Action by full commit SHA** (`actions/checkout`, `actions/setup-dotnet`, `softprops/action-gh-release`). Closes HIGH-3.
3. **Pin `wix --version 5.0.2`** exactly. Closes HIGH-4.
4. **Replace `<Files Include="stage\**" />`** in `installer/installer.wxs:42` with an explicit `<File>` list + a CI step that fails if `installer\stage\` contains anything off-allowlist. Closes HIGH-5.

**Defense-in-depth tier:**
5. **`e.Source` validation** at the top of both `WebMessageReceived` handlers. One-if-statement defense against future renderer XSS. Closes MEDIUM-1.
6. **Host allow-list in `OpenInDefaultBrowser`** so a compromised renderer can't phish via the host's `Process.Start`. Closes MEDIUM-3.
7. **`dependabot.yml`** for `nuget` and `github-actions`, plus a **`NuGet.config`** with `packageSourceMapping` pinned to `nuget.org`. Closes MEDIUM-10.
8. **`<Deterministic>true</Deterministic>`** + `actions/attest-build-provenance` so users can independently verify a release came from the public repo. Closes L-13.

Everything else from Sections 1-4 (CRITICAL-2, HIGH-1, HIGH-2, HIGH-6, MEDIUM-2/4/5/6/7/8/9, most of the L-list) requires either same-user code execution or is reliability hygiene, and is therefore deferrable under this threat model.

## Methodology note

Six general-purpose attack subagents were spawned in parallel (Opus). Each was scoped to one surface, given the local repo path, and instructed to read the red-team skill's `subagent.md`, `guardrails.md`, `attack-surfaces.md`, `tooling.md`, and `report-format.md` references before attacking. All findings were deduplicated across teams. The MSI-retry vector was independently identified by three subagents (settings, process, update path) and is reflected once here as CRITICAL-2.

No active exploitation was performed. No network requests were made to GitHub, YouTube, or any external host. All evidence is from local source.
